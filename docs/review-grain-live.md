## Findings

1. **P1 ‚Äî Marketing-plan RPC payload can never satisfy migration 0012** ‚Äî `src/data/SupabaseGrainDataGateway.ts:27-30`; `supabase/migrations/0012_grain_live_support.sql:101-132`. The gateway removes `farm_id` from every target, but the function requires each JSON target to contain a matching `farm_id`. Every non-empty target save/template replacement therefore throws; only empty-plan deletion can work. Send the complete normalized target rows, including server-bound `farm_id`, while omitting only timestamps.

2. **P1 ‚Äî SECURITY DEFINER RPC bypasses 0008‚Äôs grain-read privacy** ‚Äî `supabase/migrations/0012_grain_live_support.sql:15,41-43,313-321`; `supabase/migrations/0002_module1_rls.sql:25-38`; `supabase/migrations/0008_employee_privacy.sql:18-39`. `can_edit_farm` includes workers, while `can_read_private_financials` may deny them. Because the RPC is `SECURITY DEFINER`, an otherwise denied worker can call it and receive canonical marketing-plan rows, bypassing table RLS. Require `can_read_private_financials(p_farm_id)` before returning private rows‚Äîor tighten RPC authorization to roles allowed to read Grain‚Äîand add a denied-worker SQL test.

3. **P1 ‚Äî Grain queue rejects its own production, contract, and plan entries** ‚Äî `src/data/grainWriteQueue.ts:20-27`. `base()` first requires the complete row‚Äôs keys, then calls `scope(value)`, whose `exact()` check requires that same object to contain only the five scope keys. Consequently every production, contract, and target row fails parsing after persistence. Offline saves can write bytes and then report an error, leaving the queue blocked. Split scope-field validation from exact-object validation, validate entries before writing, and prove all four entry kinds round-trip.

4. **P1 ‚Äî Saves can be confirmed without confirming the intended canonical row/set** ‚Äî `src/data/SupabaseGrainRepository.ts:55,57,60,62`; `src/data/QueuedGrainRepository.ts:24,32`. Plain upserts do not require the returned ID and complete scope to match the submitted row. Plan replacement accepts an empty, incomplete, duplicated, or differently identified canonical set as success so long as returned rows are in the scope. Replay then removes the queue head. Require exact submitted ID/scope for single rows and exact final ID/month/count equality for plan replacements before resolving or removing queued work.

5. **P1 ‚Äî Grain can falsely report ‚ÄúAll changes synced‚Äù while Fields work is pending** ‚Äî `src/data/syncStatus.ts:7-15`; `src/data/QueuedGrainRepository.ts:17,24,32`; `src/data/QueuedFieldsRepository.ts:43-45,79-82`; `src/App.tsx:108-111,122`. Fields and Grain overwrite one global status and one global retry callback. Grain is constructed last, so ‚ÄúTry again‚Äù only replays Grain; either empty replay can overwrite the other module‚Äôs pending/blocked state with `synced`. Aggregate per-module states and retry actions so ‚Äúsynced‚Äù is emitted only when both queues are empty.

6. **P1 ‚Äî 0008-denied users receive a dishonest empty-Grain state** ‚Äî `src/data/SupabaseGrainDataGateway.ts:13-23`; `src/data/SupabaseGrainRepository.ts:41-52`; `src/GrainModule.tsx:41-46`. RLS denial normally appears as successful empty arrays, not an error. The repository does not check `can_read_private_financials`, so the UI tells a denied member to add a crop assignment instead of explaining that Grain records are private. Add an explicit permission probe and show a calm farmer-English access message before interpreting empty private tables as real emptiness.

7. **P1 ‚Äî The live flip strands farms without an existing production estimate** ‚Äî `src/data/backends.ts:1-5`; `src/GrainModule.tsx:45-46`. Grain is already switched to Supabase, but an empty production-estimate table exposes no editor and incorrectly directs the user to Fields‚Äîeven when valid crop assignments already exist. Implement the design‚Äôs first-estimate editor from live crop assignments or keep Grain behind the release gate until real estimates are provisioned.

8. **P1 ‚Äî USDA ‚Äúdisplay-only‚Äù basis enters position and revenue math** ‚Äî `src/GrainModule.tsx:27,79`; `src/components/MarketQuote.tsx:104`. `latestBasis()` selects every cash-bid row, including `[USDA MARS 2850]`, and that value feeds partial and planned revenue. This contradicts the UI promise that calculations use basis the farmer entered and the requested display-only boundary. Separate manual bids from feed observations for calculations; retain MARS rows only in history and alert evaluation.

9. **P1 ‚Äî Enterprise-scoped production is reconciled as the entire entity/commodity acreage** ‚Äî `src/data/SupabaseGrainRepository.ts:33,52,55`. Reconciliation filters crop year, commodity, and optional operating entity but ignores `enterprise_label`. Multiple enterprise-scoped estimates therefore each receive the same full acreage and expected bushels. Either establish an authoritative Fields mapping for enterprise labels and filter it exactly, or reject non-null enterprise production scopes until such a mapping exists.

10. **P2 ‚Äî Owner email delivery is absent and failures are completely silent** ‚Äî `src/data/grainAlerts.ts:23-30`; `src/GrainModule.tsx:41`. The client invokes `deliver-grain-alert`, but no `supabase/functions` implementation exists. Authentication and delivery failures are returned or discarded without logging, and overlapping refreshes can invoke the same unsent key concurrently before local dedup is recorded. Add the reviewed Edge Function with owner verification, canonical re-reads, fixed recipient, sanitization, deterministic-key throttling and operational logging; serialize in-flight keys client-side.

11. **P2 ‚Äî Date and timestamp mappers accept normalized or noncanonical values** ‚Äî `src/data/SupabaseGrainRepository.ts:16-19,31`; `src/data/grainWriteQueue.ts:16-17`. `Date.parse` accepts impossible dates such as `2026-02-31` by normalizing them, and the timestamp check accepts broadly parseable strings rather than canonical timestamps. Validate calendar components by exact round-trip and require an explicit ISO timestamp shape.

12. **P2 ‚Äî MARS outage/staleness is not represented honestly** ‚Äî `src/GrainModule.tsx:93`. The UI displays a last date but never determines whether the observation is stale and never shows ‚ÄúBasis feed unavailable ‚Äî last updated ‚Ä¶‚Äù. Old MARS rows continue to look like ordinary history indefinitely. Calculate age/provenance separately and show the required stale/unavailable state without inventing a new observation date.

13. **P2 ‚Äî The Grain regression suite does not exercise the contract it claims to cover** ‚Äî `src/data/SupabaseGrainRepository.regression.ts:34-55`. The ‚Äú11-15‚Äù section never constructs `QueuedGrainRepository` or `GrainWriteQueue`, so FIFO replay, all entry kinds, cross-tab behavior, context isolation, optimistic overlays, canonical-response confirmation, old-key isolation, blocked classification, privacy denial, and unknown-commit replay are untested. This is why the self-rejecting queue parser passed review. Implement the design‚Äôs fifteen network-free checks rather than grouping a few assertions under those numbers.

14. **P2 ‚Äî Repository/RPC percentage semantics and migration verification differ** ‚Äî `src/data/SupabaseGrainRepository.ts:51,60`; `supabase/migrations/0012_grain_live_support.sql:228-230,333-341`. The repository rejects totals above `100.000001`, while 0012 allows through `100.0001`; the SQL acceptance cases are comments, not executable tests. Use one shared tolerance matching the migration and add executable tests for authorization, nullable scopes, replay, empty replacement, omitted deletion, and rollback.
